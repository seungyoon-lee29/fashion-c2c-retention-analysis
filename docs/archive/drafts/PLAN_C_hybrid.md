# 후보 C — 하이브리드 (B 서사 겉 + A 식별지도 깊이) ★추천★
> **패션 C2C 그로스 분석 — 퍼널·리텐션을 읽고, "보이는 레버가 진짜인지" 인과로 검증하고, 다음 실험을 설계한다 (DA 포트폴리오)**

## 정체성
"행동로그로 **퍼널 누수를 찾고, 리텐션 드라이버를 검증하고, 다음 실험을 설계하는** 프로덕트 데이터 분석가.
*겉보기 좋은 지표(카트·찜)에 속지 않고* 진짜 북극성(재방문)을 움직이는 행동만 골라낸다."

> 설계 원칙: **겉(60초 스캔)은 DA-legible, 속(딥다이브)은 인과 엄밀성.** 퍼널·코호트는 누구나 그리지만,
> 그 위에 **식별 가능성 지도 + 실험 설계**까지 얹는 게 이 작품의 해자다.

## 데이터 (단일 출처)
MerRec(Mercari C2C 행동로그) 5파티션·277만 이벤트·43k 유저·49만 세션·31일. 가입 필드 없음(→ t0=첫 관측).

## 분석 척추 (4단계 — 앞 2개=온램프, 3·4=해자)

### 1. 퍼널 — "어디서 새는가" (온램프)
- 도달률 view 98.7 → like 52.6 → cart 22.9 → offer 17.1 → **buy 4.4%**. 단계별 전환·절대 드롭오프 + 세그먼트별(카테고리·세션길이·신규 vs 재방문).
- 산출 `docs/funnel_report.md`. **인사이트**: 전환 극희소(4.4%) → 전환을 KPI로 쓰면 underpowered. (이게 3단계 복선.)

### 2. 코호트 리텐션 — "누가 돌아오는가" (온램프)
- 첫 관측일 코호트 × 재방문(D1·D3·D7) 매트릭스 + 히트맵. 드라이버 = **인게이지먼트 폭**(세션수·활동일수·조회수), 단일 행동 아님(SHAP).
- 산출 `docs/cohort_report.md`. **인사이트**: 리텐션은 "한 번의 결정적 행동"이 아니라 폭에서 온다.

### 3. 식별 가능성 지도 — "보이는 레버가 진짜 레버인가" ★핵심 해자★
- 카트=리텐션 레버처럼 보임(카트 유저 재방문 47.7% vs 32.7%, **+15pp 연관**). 인과 보정(g-formula/IPTW)에서 **~0 붕괴**.
- **카트 단독이 아님**: 레버 4종(cart·offer·탐색폭·like≥3) **전부 positivity 실패**(e_max→1.0, SMD 0.37–1.21).
  - `like≥3` = **교과서급 함정 figure**: naive +25.7pp가 IPTW 후에도 +25.0pp → 순진하면 "강건한 인과"로 오독.
    실제론 propAUC=1.00·극단가중으로 **positivity가 깨져 보정이 무의미**한 것. ("보정 후에도 안 변하면 오히려 의심.")
- 머신 = v1 이산시간 경쟁위험 해저드(`survival.py`) + g-formula/IPTW/MSM(`impact.py`) + E-value 봉투. (X=cart는 `aha_cart`로 이미 인코딩 → 머신 수정 0.)
- 산출 `docs/causal_report.md` (← throwaway 스크립트 `version2/identifiability_map.py`로 커밋, `make`로 재현). **결론**: "이 데이터로 식별 가능한 리텐션 레버는 없다 ⇒ 실험 필수."

### 4. A/B 설계 — "그럼 뭘 실험하나" (결론, 단정이 아니라 설계)
- 식별 불가가 *결론* → 답은 실험으로만. 가설(카트-넛지 → 재방문) · 랜덤화 단위(유저) · 주지표(D7 재방문) ·
  **가드레일 + SRM(표본비 불일치) 체크 + MDE/표본수 표**(측정 base rate 기반).
- 산출 `docs/ab_test_design.md`. **인사이트**: 분석가의 일은 "정답 단정"이 아니라 *올바른 실험 설계*.

## 방법론 부록 (깊이·범위 증명 — 헤드라인 아님)
- **추천 폭(범위 증명)**: 세션 prefix → 다음행동/카테고리 예측, POP→**마르코프**(c1 0.59→0.85 = "세션 직전 신호 > 인기"). GRU4Rec·SASRec·item-level은 *연구 노트*로만(C2C 73% 1회성 콜드스타트). **여기서 멈춤(YAGNI).**
- **OPE(차별화 한 문단)**: 합성에선 로깅정책 설계로 IPS/DR 무편향 *증명* 가능 / 실데이터는 노출 propensity 부재로 식별 불가 — "노출 무작위성(A/B) 주어지면 다음 단계". *코드 미구현, 서술만.*
